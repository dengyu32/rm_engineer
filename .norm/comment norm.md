### comment norm
> 约定速成的注释规范

#### 项目规范
> 参考 teleop_node 的注释风格
> 目标：结构清晰、注释短句、与代码层级对齐

标准结构：
`文件头分组注释` + `模块块注释` + `成员分区注释` + `必要行内注释`

hpp:
- include 之前先写依赖分组注释，使用 `// Rely`
- include 分组标签统一使用 `//< Xxx`
- 类定义前使用三段式块注释：
~~~cpp
// ============================================================================
//  ClassName
// ----------------------------------------------------------------------------
//   - 职责 1
//   - 职责 2
// ============================================================================
~~~
- 类内函数/成员按功能分区，使用 `//<`：
~~~cpp
//< ROS 2 Topic Callbacks
//< Internal Func
//< Topic Params
~~~
- 行内注释仅用于“状态约束/共享资源/易错点”，避免解释显而易见代码

cpp:
- 关键模块入口使用块注释分隔（ctor/callback/internal/main）：
~~~cpp
// ============================================================================
//  ctor
// ============================================================================
~~~
- 回调和内部函数分段命名保持短且直接，如 `ROS 2 Topic Callbacks`、`Internal Func`
- 注释说明“做什么/为什么”，不重复“怎么做”
- 异常路径（参数非法、模型加载失败、回退策略）应有注释或清晰日志

命名与语言:
- 允许中英混合，但一条注释尽量只表达一个意思
- 术语优先与模块一致：`teleop` `collision` `fallback` `joint`
- 避免大段自然语言；优先短句和项目内固定词

禁止项:
- 禁止无信息注释（如“赋值”“进入函数”）
- 禁止注释与代码语义不一致
- 禁止装饰性注释过密，连续空注释块应删除
